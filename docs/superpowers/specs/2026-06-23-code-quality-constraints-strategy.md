# 代码质量约束策略（风格 / 正确性 / 架构）

> 生成日期：2026-06-23
> 来源：苏格拉底式探讨——"风格、逻辑性、架构设计该用什么约束 AI"
> 性质：**策略/指导框架**（不是单一特性的实现 spec）。宪法/analyze 是其中一层。
> 关联：[constitution](./2026-06-23-opsx-constitution-design.md)、[probe](./2026-06-23-opsx-probe-design.md)

---

## 0. 核心命题

约束 AI 的代码质量，不存在单一机制。**该用什么机制，由两根轴决定**：

1. **它可机械判定吗？**（能否写成确定性规则，不依赖任何人主观）
2. **在哪个阶段拦它最便宜？**（写代码前 / 写完后 / 评审时 / 长期）

```
可机械判定            → 工具（formatter / linter / type / test / dep-linter）
不可机械、设计期最便宜 → plan 级原则，analyze 在写代码前审
不可机械、只在代码现形 → 快反馈 + 可逆 + 传感器 + 回流 + 人（见第 4 节）
正确性               → 可执行测试（per-case 机械门），由 spec 驱动
```

**第一原则：能交给确定性工具的，绝不交给"写条原则让 AI 自觉"。** 措辞没有约束力，只有独立的机械核对者才有。

---

## 1. 三类信息各归各的机制

| 类别 | 主机制 | 不该用 |
|---|---|---|
| **代码风格**（格式/命名/惯用法） | formatter + linter + 类型系统；残余靠"对齐周围代码" | 写进宪法 / 靠 AI 判断（最大浪费——它最可机械化） |
| **代码正确性**（逻辑/边界） | **spec 定义对错 + 测试钉死契约**（TDD：测试是不依赖评审时主观的 per-case 门）+ 类型 + 让它真跑给你看 | 写"要正确"这种原则（不可核对；正确性逐函数不同） |
| **代码架构**（边界/依赖/抽象） | 一半工具（依赖规则）+ 一半**设计期判断**（analyze 前置）+ 残余人类 review | 指望一个机制全包 |

AI 的三个典型坏味及其克星：
- **看似对实则微妙错** → 必过的测试 + 类型 + 跑给你看
- **过度设计 / 无谓抽象** → 设计期"简单优先"原则 + design review 前置 + "对齐现有模式"
- **风格不一** → formatter/linter，非协商、自动化

---

## 2. 分层防御（不要让一个机制管所有事）

每层拦下层拦不住的：

| 层 | 机制 | 管什么 | 何时 | OpenSpec 现状 |
|---|---|---|---|---|
| 1 | 类型 / 编译器 | 一类逻辑错 | 持续 | 已有（TS） |
| 2 | formatter + linter + 依赖 linter | 风格、可机械化的架构边界 | 保存/提交 | 部分（无 dep-linter） |
| 3 | 测试 | 正确性 | CI + **apply-TDD** | 已有；apply-TDD 设计见 [opsx-apply-tdd-design](./2026-06-23-opsx-apply-tdd-design.md) |
| 4 | 宪法 via `/opsx:analyze` | 设计期的架构/行为原则 | 写代码前 | **本批设计新增** |
| 5 | 人类 review | 品味、novel 架构、问责 | PR | 外部 |

## 3. 架构判断前移到设计期

### 3.1 真正的瓶颈：架构够不够"显式到能被判断"

AI 的架构烂，多数不是"做错决定"，而是**决定从没被显式做出来——从代码里悄悄长出来**。所以前移第一步**不是审判，是逼显式化**：design 不写架构承诺，analyze 无从审起。

### 3.2 逼 design 显式化（design 模板的强制章节）

1. **单元与边界**：改动哪些模块，每个单一职责一句话
2. **依赖方向**：谁依赖谁；明确"X 不得依赖 Y"
3. **新抽象清单 + 驱动需求**：每个抽象点名它服务的**本 change** 哪条需求
4. **备选方案 + 否决理由**（ADR 式）：≥2 个候选 + 为什么否决
5. **与现有架构的关系**：沿用哪个模式；偏离则说明为什么

### 3.3 两个强迫装置（逼 AI 露马脚）

**A. 每个抽象必须点名当前驱动需求**（掐死过度设计）
> `[judgment]`：design 里每处抽象/分层/泛化都能指向本 change 的具体需求；凡用"未来/可扩展/灵活/可插拔"辩护且无当前驱动 → CRITICAL。

**B. 备选方案 = 逼它展示思考过程**
> `[structure]`：含"备选方案"章节，≥2 候选 + 否决理由
> `[judgment]`：否决理由是否实质（指出具体权衡），还是橡皮图章。

### 3.4 analyze 能在 design 文本里抓的"架构 tell"

| design 里的信号 | 问题 | 类型 |
|---|---|---|
| "flexible/future/可配置"无当前驱动 | 投机性泛化 | judgment |
| 小 change 列出多个新模块/层 | 复杂度与范围不成比例 | structure+judgment |
| 一个组件塞多个不相干职责 | God component | judgment |
| 引入新机制而现有 specs 已有等价物 | 重复造轮子 | judgment |
| 直接跳进实现细节、不写边界 | 决策没显式化 | structure（打回） |

### 3.5 比例性交叉检查（设计期独有）

analyze 同时握有 **change scope**（proposal）和**架构足迹**（design）：
> `[judgment]`：架构足迹与 change 范围成比例吗？1 需求配 4 抽象 = CRITICAL。

代码期丢了"意图"这个锚，做不到这条。

## 4. 前移不了的怎么办：从"阻止"转向"早暴露 + 改得起 + 可纠"

有一类架构错（抽象到底对不对、涌现复杂度、集成摩擦、品味）**只有建出来才知道**。对它放弃"事前阻止"，改用六招：

1. **压缩反馈延迟**：先做最薄纵切，**新抽象的第一个真实调用者/测试就是传感器**；"用起来别扭"= 抽象错了，立刻回改 design，别推到 3000 行。
2. **让改错便宜（与"简单优先"复利）**：设计期没预盖投机框架 → 发现要换抽象时，改的是小而具体的东西，不是拆错框架。少承诺 → 晚发现也不疼。
3. **建成后的机械传感器（fitness functions）**：依赖环（dep-linter）、圈复杂度、耦合度、文件膨胀、改动热点，放 CI。不判"对不对"，只报警"在恶化"。把"前移不了"的一部分**后移到持续监测**。
4. **暴露后回流进 design，不默默打补丁**：实现发现边界错 → 改代码**同时**改 design + 重跑 analyze（避免 Flow-Back 静默分歧）。OpenSpec 已有钩子（`apply-change.ts:82`），缺的是把它变成**不可跳过且大声**。
5. **长期烟雾探测器：waiver / design 修订日志**：同一原则在同一模块反复被豁免，或同一处 design 反复被修订 = 强信号（原则错或架构错）→ 起一个 **refactor change**（OpenSpec 里 refactor 也只是另一个 change）。把"只有时间能暴露的架构错"变成可观测、可触发的信号。
6. **人类 review 只留给不可约简的内核**：品味、无先例的 novel 架构、后果大/不可逆的决策。前五层把可机械、可前移、可传感的都剥掉，让人只在这三处花注意力，且手握 design + 备选方案。

### 诚实的终点

**永远做不到"零错误架构"，目标也不是它。** 目标是把错误架构从"建完很久、代价巨大、无人知晓"变成：

> **早暴露（1、3）+ 改得起（2）+ 留痕可触发（5）+ 回流不漂移（4）+ 人只管内核（6）。**

不可消灭，但可以让它**廉价、可见、可纠**。

---

## 5. 与 constitution / probe 的关系

- **constitution + `/opsx:analyze`** = 第 2 节的**第 4 层**，承担"架构/行为原则的设计期判断"（第 3 节）。它是本策略的一个组件，不是全部。
- **probe** = 在 propose 前把需求/范围挖清楚，给"比例性交叉检查"（3.5）提供可信的 scope 锚。
- 其余各层（工具、测试、CI fitness functions、人类 review）**不属于宪法机制**，是平行的独立机制——这正是"不要让一个机制管所有事"的体现。

## 6. 落到 OpenSpec 的抓手清单

| 招 | 抓手 | 状态 |
|---|---|---|
| 风格 | formatter + ESLint 规则 | 已有/可加 |
| 可机械化架构边界 | dep-linter（import-linter / eslint-plugin-boundaries） | **建议新增** |
| 正确性 | 测试 + apply-TDD 循环（"done = 测试绿"） | **本批设计**（opsx-apply-tdd-design） |
| 设计期架构判断 | design 模板强制章节（3.2）+ 宪法架构判据（3.3-3.5）+ analyze | **本批设计** |
| 建成后传感 | CI fitness functions（环/复杂度/热点） | **建议新增** |
| 回流不漂移 | apply 的"design issue → 更新 artifacts"钩子变强制 | 钩子已有，待强化 |
| 长期信号 | waiver / design 修订日志 → refactor change | waiver 待实现 |
| 内核 | 人类 PR review | 外部 |

> 注意：只有"人类 review"一项真要靠人，其余大多能落到机制/工具上。"前移不了"≠"只能靠人盯"。
